## What is zot-answer?

[zot-answer](https://github.com/patriceckhart/zot-answer) is a TypeScript extension for the [zot](https://www.zot.sh) CLI tool. It registers a `/answer` slash command that extracts numbered questions from the last assistant message and opens an interactive terminal panel where you type answers. The extension communicates with its host through a JSON frame protocol over stdin/stdout.

It's a single file. No package.json, no build system, no framework. Just 236 lines of TypeScript and an `extension.json` manifest that tells zot to run it with `npx tsx index.ts`.

## Why containerize a CLI extension?

Not every project that goes through a PoC pipeline is a web service or an ML model. CLI extensions, developer tools, and protocol-based utilities are common in real engineering workflows. Proving they can run in containers validates something useful: your platform handles more than HTTP endpoints.

For this PoC, we wanted to answer two questions. Can a stdin/stdout TypeScript extension be containerized with a UBI base image? Can its JSON protocol behavior be validated through Kubernetes Jobs?

## Containerizing with UBI Node.js

The extension has no `package.json` and depends only on `tsx` for TypeScript execution. We built a `Dockerfile.ubi` that creates a minimal package manifest inline and installs tsx as the sole dependency:

```dockerfile
FROM registry.access.redhat.com/ubi9/nodejs-22

WORKDIR /opt/app-root/src

RUN echo '{"name":"zot-answer","version":"0.1.0","type":"module","dependencies":{"tsx":"^4"}}' > package.json
RUN npm install --production

COPY index.ts .
COPY extension.json .

USER 0
RUN chgrp -R 0 /opt/app-root && chmod -R g=u /opt/app-root
USER 1001

ENTRYPOINT ["npx", "tsx", "index.ts"]
```

Two things to note for OpenShift compatibility. First, files added by `COPY` are owned by root and the default non-root user can't `chgrp` them. Switching to `USER 0` for the permission fix, then back to `USER 1001`, resolves this. Second, there's no `EXPOSE` directive because the extension doesn't listen on any port. It reads JSON from stdin and writes JSON to stdout.

The build ran on OpenShift using a binary BuildConfig and pushed the image to `quay.io/aicatalyst/zot-answer:latest`.

## Deploying as Kubernetes Jobs

Since the extension isn't a long-running service, we used Kubernetes Jobs instead of Deployments. Each Job pipes JSON frames to the container's stdin and captures the stdout output.

Here's the startup validation Job:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: zot-answer-startup
  namespace: poc-zot-answer
spec:
  backoffLimit: 0
  activeDeadlineSeconds: 120
  template:
    spec:
      containers:
        - name: zot-answer
          image: quay.io/aicatalyst/zot-answer:latest
          command: ["/bin/sh", "-c"]
          args:
            - |
              printf '{"type":"hello_ack"}\n{"type":"shutdown"}\n' \
                | npx tsx /opt/app-root/src/index.ts
      restartPolicy: Never
```

The Job sends a `hello_ack` (acknowledging the extension's hello) followed by a `shutdown` command. The extension responds with its registration frames and exits cleanly.

## Running the PoC tests

We ran two scenarios against the deployed Jobs.

**Scenario 1: Protocol handshake.** The extension starts, sends its `hello` frame with capabilities, registers the `/answer` command, subscribes to events, sends `ready`, receives `hello_ack`, then responds to `shutdown` with `shutdown_ack` and exits with code 0.

**Scenario 2: Question extraction.** We sent a `command_invoked` frame with the args `"1. What is AI? 2. What is ML? 3. What is DL?"`. The extension responded with a `command_response` containing an `open_panel` action. The panel included the extracted question text and an answer input field.

| Scenario | Status | Duration |
|----------|--------|----------|
| container-startup | PASS | 0.2s |
| question-extraction | PASS | 0.21s |

Both scenarios passed. The container correctly preserved the extension's protocol behavior and question-extraction logic.

## What we learned

**OpenShift builds need a file named `Dockerfile`.** The initial build failed because we named our file `Dockerfile.ubi`. OpenShift's binary build strategy looks for `Dockerfile` by default. Copying the file solved it, but it's worth knowing if you standardize on alternative Dockerfile names.

**UBI Node.js images work well for minimal TypeScript projects.** The `ubi9/nodejs-22` image includes npm and Node.js. Adding tsx as a runtime dependency took 2 seconds and installed just 3 packages. The resulting image is lean and functional.

**Jobs are the right primitive for CLI tools.** Deploying a stdin/stdout extension as a Deployment would cause CrashLoopBackOff because the process exits after handling its input. Jobs with `restartPolicy: Never` match the execution model perfectly.

**Quay repos default to private.** The initial deploy hit `ErrImagePull` because the newly created Quay repository was private. Making it public via the API resolved the issue. For production workloads, you'd use an image pull secret instead.
